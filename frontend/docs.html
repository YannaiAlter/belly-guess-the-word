Belly Docs

Complete developer reference for the Belly smart contract and frontend.

Overview

Belly is an on-chain word-guessing game built on IGRA Network (an EVM-compatible Kaspa Layer 2). Players pay 10 iKAS per guess. The first player to guess the secret word wins 95% of the accumulated prize pool; the remaining 5% is the operator fee.

The secret word is never transmitted to the blockchain. Instead, the owner computes keccak256(secretWord) client-side and stores only the hash. When a player submits a guess, the contract hashes their plaintext guess and compares it to the stored hash.

Key properties: Trustless win verification · No plaintext exposure · Reentrancy-safe · 1-day timeout mechanism

Quick Start

Get a local instance running in under 5 minutes.

Prerequisites

  • Node.js ≥ 18
  • npm ≥ 9
  • A browser with MetaMask or KasWare installed (or use Dev Mode)

Steps

  1. Install dependencies 
    npm install
  2. Start a local Hardhat node (keep this terminal open) 
    npx hardhat node
  3. Compile & deploy the contract 
    npm run deploy
    This writes frontend/contract.json containing the contract address and ABI.
  4. Serve the frontend 
    npm run serve
    Open http://localhost:3000 and connect with Dev Mode.

Architecture

The project consists of three layers:

  • Smart contract (contracts/WordGuessing.sol) — Solidity 0.8.20, deployed to IGRA Network (or Hardhat for dev).
  • Frontend (frontend/) — Plain HTML + vanilla JS using ethers.js v6. No build step required.
  • Toolchain — Hardhat for compilation, testing, and deployment. serve for static hosting.

frontend/contract.json is auto-generated at deploy time and contains the contract address, ABI, network info, and deploy timestamp. It is the only coupling between the contract and the frontend.

Functions

owner only state-changing startGame(bytes32 _wordHash, string _hint)

Opens a new game round. The _wordHash must be computed off-chain as:

// JavaScript (ethers.js v6)
const wordHash = ethers.keccak256(ethers.toUtf8Bytes(secretWord));
// Solidity (on-chain utility)
wordGuessing.computeHash("blockchain"); // returns bytes32

Emits: GameStarted

payable state-changing submitGuess(string guess)

Submits a guess. Must attach exactly GUESS_PRICE (10 iKAS) as msg.value. Checks performed (in order): game active · correct payment · cooldown elapsed · guess limit not reached.

The contract hashes the plaintext guess and compares with wordHash. On a correct guess, 95% of the pot is sent to the caller immediately; 5% to the owner.

Emits: GuessSubmitted (always) + WinnerDeclared (on correct guess)

// ethers.js v6 — submit a guess
const tx = await contract.submitGuess("blockchain", {
  value: ethers.parseEther("10")
});
const receipt = await tx.wait();

owner only state-changing resetTimedOutGame()

Allows the owner to close the game and reclaim the pot once GAME_TIMEOUT (1 day) has elapsed without a winner. Protected by nonReentrant.

Emits: GameReset

view getGameState()

Returns a full snapshot of the current game. This is the primary read call used by the frontend.

(bool    isActive,
 bytes32 wordHash,
 string  hint,
 uint256 potBalance,
 address winner,
 uint256 guessesCount,
 uint256 gameStartedAt,
 uint256 timeRemaining) = await contract.getGameState();

view getPot()

Returns the current pot balance in wei.

pure computeHash(string word)

Returns keccak256(abi.encodePacked(word)). Useful for testing or verifying a hash on-chain. Use this off-chain only — calling it reveals the word in the transaction trace.

Events

EventParametersEmitted when
GameStarted bytes32 wordHash, string hint, uint256 timestamp Owner calls startGame()
GuessSubmitted address player, uint256 totalGuesses, bool correct, uint256 timestamp Any player submits a guess (correct or not)
WinnerDeclared address winner, uint256 winnerAmount, uint256 feeAmount A guess is correct and funds are distributed
GameReset address owner, uint256 reclaimedAmount, uint256 timestamp Owner resets a timed-out game

Listening to events (ethers.js v6)

contract.on("GuessSubmitted", (player, totalGuesses, correct, timestamp) => {
  console.log(`${player} guessed — ${correct ? "correct!" : "wrong"}`);
});

contract.on("WinnerDeclared", (winner, winnerAmount, feeAmount) => {
  console.log(`Winner: ${winner}, payout: ${ethers.formatEther(winnerAmount)} iKAS`);
});

Frontend Integration Guide

The frontend uses ethers.js v6 loaded from a CDN. There is no build step — the page is a single index.html and app.js.

Loading the contract

At deploy time, scripts/deploy.js writes frontend/contract.json with the contract address and full ABI. The frontend fetches this file on load: 

const res  = await fetch("./contract.json");
const info = await res.json(); // { address, abi, network, chainId, deployedAt }

const contract = new ethers.Contract(info.address, info.abi, signer);

Reading state

Call getGameState() for a single-RPC snapshot of all game state. Poll it on a timer (every 5–10 seconds) and subscribe to events for real-time updates.

Hashing the secret word

The owner hashes the word client-side before calling startGame:

const wordHash = ethers.keccak256(ethers.toUtf8Bytes(secretWord));
await contract.startGame(wordHash, hint);

Important: Never log or transmit the plaintext secret word. Only send the hash to the chain. Exposing the secret in the browser console or network requests makes the game trivially solvable.

Wallet Support

The frontend supports three connection modes:

ModeProviderUse case
KasWare window.kasware.ethereum Recommended for IGRA Network mainnet
MetaMask window.ethereum (isMetaMask) General EVM wallet; works on any chain
Dev Mode JsonRpcProvider on http://127.0.0.1:8545 Local Hardhat — no browser extension needed

Provider detection is done once on DOMContentLoaded. Both KasWare and MetaMask fire accountsChanged and chainChanged events which trigger a reconnect or page reload respectively.

Deploying to IGRA Network

  1. Create a .env file from the template 
    cp .env.example .env
    Fill in PRIVATE_KEY, IGRA_RPC_URL, and IGRA_CHAIN_ID.
  2. Install dotenv (if not already present) 
    npm install --save-dev dotenv
  3. Deploy to IGRA mainnet 
    npm run deploy:igra
  4. Serve the frontend

    Upload frontend/ to any static hosting (Vercel, Netlify, IPFS, etc.). No server required.

Security: Never commit your .env file. Add it to .gitignore before pushing to a public repo.

Security Patterns

Checks-Effects-Interactions (CEI)

In submitGuess, all state mutations occur before any .call{value:…}() transfers. This prevents reentrancy even in the absence of a mutex.

Reentrancy guard (nonReentrant)

A boolean mutex (_locked) is used as a second line of defence on submitGuess and resetTimedOutGame. The CEI pattern is the primary protection; the guard adds depth.

Hash-based secret verification

The secret word is never transmitted on-chain. Only its keccak256 hash is stored. A player's plaintext guess is hashed inside the EVM and compared. This is tamper-proof: the owner cannot change the target word mid-game without triggering a new transaction.

Known limitation: Short or common words are susceptible to brute-force guessing. Owners should choose uncommon, multi-syllable words. A dictionary-attack resistant scheme (e.g. salted hashes) is a possible future enhancement.

Timeout & drain prevention

If no winner is found within GAME_TIMEOUT (1 day), the owner can call resetTimedOutGame() to reclaim the pot. This prevents funds being locked forever if all players give up.

Per-player limits

A cooldown of 10 seconds between guesses and a cap of 20 guesses per address per game prevent spam and keep the game fair for all participants.

The contract has been written conservatively and is not audited. For a high-value production deployment, a professional security audit is strongly recommended.
